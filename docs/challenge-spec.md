# Спецификация задания

## Кратко

| Блок | Определение |
|---|---|
| Ключевая задача | Разработать универсальный конвейер сжатия и восстановления гидродинамических моделей |
| Вход | Каталог исходной модели |
| Выход | Стандартизированный сжатый артефакт и восстановленная модель |
| Главная цель | Баланс коэффициента сжатия, точности восстановления, структурной целостности и времени выполнения |
| Критичное ограничение | Полная автоматизация и воспроизводимость в среде проверки |

## Постановка задачи

Нужно разработать универсальный алгоритм сжатия и восстановления гидродинамических моделей.

Решение должно корректно работать на моделях с различающимися:

- размерами сетки;
- числом временных шагов;
- наборами статических свойств;
- наборами динамических свойств;
- структурой вспомогательных файлов проекта.

## Что получают участники

| Материал | Назначение |
|---|---|
| Публичный датасет | Локальная разработка и публичная оценка |
| Стартовый набор | Быстрый вход в структуру данных и требования к решению |
| Базовое решение | Минимально валидная эталонная реализация |
| Публичный оценщик | Локальная быстрая проверка и публичная оценка |
| Шаблон подачи решения | Эталонная структура репозитория решения |
| Примеры | Примеры артефактов, метаданных и журналов |

## Что должно делать решение

Решение должно:

1. Принять каталог модели на вход.
2. Построить сжатый артефакт.
3. Восстановить модель из этого артефакта.
4. Пройти автоматическую проверку без ручных действий.

Точный интерфейс командной строки и обязательная структура репозитория заданы в [submission.md](./submission.md).

## Что считается сильным результатом

Сильное решение одновременно:

- значительно уменьшает объем модели;
- сохраняет корректную структуру проекта;
- восстанавливает как можно больше полезной информации;
- работает без ручных действий;
- проходит проверки в стандартной среде организаторов.

## Ключевые требования

- Полная совместимость с интерфейсом командной строки.
- Отсутствие графических зависимостей.
- Отсутствие сетевого доступа во время работы в среде проверки.
- Корректная работа в чистом окружении.
- Обработка модели целиком, а не отдельных фрагментов.
- Предсказуемый и стандартизированный результат.

## Что будет оцениваться

| Семейство метрик | Что измеряется |
|---|---|
| Коэффициент сжатия | Насколько хорошо решение уменьшает объем модели |
| Структурная целостность | Сохраняется ли корректная структура проекта после восстановления, включая обязательные файлы и связи `INCLUDE` |
| Предметная точность | Насколько точно восстановлены свойства (допускается сжатие с потерями при погрешности чисел с плавающей точкой до 1e-5) |
| Время выполнения | Насколько быстро выполняются `compress.py` и `decompress.py` |
| Соответствие требованиям | Выполнены ли обязательные требования к решению |

## Что не является целью

- ручная правка восстановленной модели;
- настройка среды проверки под конкретную команду;
- использование нестандартного интерфейса;
- переоптимизация только под один публичный пример.

## Среда проверки

При финальной проверке решение запускается в изолированной среде:

- **Операционная система:** Ubuntu 22.04 LTS (Docker)
- **Версия Python:** 3.10
- **Сеть:** Заблокирована (`--network none`). Любое обращение к сети приведет к ошибке при подаче решения.
- **Политика допуска для чисел с плавающей точкой:** При проверке единой метрики точности восстановления допускается сжатие с потерями для чисел с плавающей точкой при относительной погрешности до `1e-5`. Сравнение текстовых deck-файлов устойчиво к комментариям, пробелам и переносам строк, но остается строгим к порядку токенов и строковым значениям. Бинарные структуры сравниваются по точному совпадению.
