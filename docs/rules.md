# Правила участия

## Краткое резюме правил

| Категория | Основное правило |
|---|---|
| Формат участия | К участию приглашаются команды от 2 до 5 человек |
| Единый источник правил состязания | Только этот репозиторий является официальным, и требования в нём являются обязательными |
| Воспроизводимость | Решение должно запускаться в чистой среде без ручных действий |
| Честная игра | Попытки обойти приватную валидацию запрещены |
| Дисциплина подачи решений | Соблюдение порядка подачи решения обязательны |

## 1. Формат участия

- К участию приглашаются команды от 2 до 5 человек.
- Одна команда использует одно название на протяжении всего хакатона.
- Каждая команда должна завести собственный github-репозиторий для публикации решений и поделиться доступом с организаторами хакатона.

## 2. Единый источник правил состязания

- Официальными считаются только правила и документы в этом репозитории.
- Ответы в чатах и устные комментарии имеют силу только после фиксации в репозитории.

## 3. Что разрешено

| Разрешено | Условие |
|---|---|
| Python и совместимые офлайн-инструменты | Должны воспроизводимо работать в среде проверки |
| Библиотеки с открытым исходным кодом | Должны быть явно указаны в зависимостях |
| Языковые модели как инструмент разработки | Не освобождают от ответственности за воспроизводимость |
| Собственные алгоритмы и модели | При соблюдении всех правил и требований к решению |
| Предобученные компоненты | Только если они воспроизводимы офлайн и поставляются корректно |

## 4. Что обязательно

- соблюдать требования к решению и утвержденный формат подачи решения;
- явно фиксировать зависимости;
- обеспечивать запуск без ручного вмешательства;
- соблюдать ограничения по числу отправок решения.

## 5. Что запрещено

| Запрещенное поведение | Почему это запрещено |
|---|---|
| Сетевой доступ из решения во время оценки | Запуск происходит в Docker-контейнере (`--network none`). Попытка обойти изоляцию приведет к отклонению отправки решения |
| Преднамеренная утечка памяти или лавинообразное порождение процессов | Нарушает работоспособность системы оценки и ведет к мгновенной дисквалификации |
| Ручные действия внутри конвейера проверки | Нарушают требование полной автоматизации |
| Попытки извлечь приватные данные или скрытую логику | Нарушают честность соревнования |
| Плагиат или копирование чужих решений | Нарушает принципы честной игры |
| Зависимость от локального состояния машины участника | Делает решение невоспроизводимым |
| Скрытая постобработка вне описанного конвейера | Искажает реальный способ работы решения |
| Подмена результатов вместо запуска `compress` и `decompress` | Нарушает смысл задания |

## 6. Требования к воспроизводимости

- Решение должно работать в чистой среде организаторов.
- Все зависимости должны быть зафиксированы или иным образом закреплены.
- Любой дополнительный артефакт, необходимый для запуска, должен быть включен в поставку решения.

## 7. Политика подачи решений

- Количество отправок решения регулируется [submission.md](./submission.md).
- Некорректные отправки решения не попадают в таблицу результатов.
- Организаторы могут отклонить отправку решения, не соответствующую обязательным требованиям.

## 8. Основания для дисквалификации

- нарушение принципов честной игры;
- попытка обойти скрытую валидацию;
- вредоносное поведение внутри конвейера проверки;
- сокрытие существенной части сценария работы решения;
- намеренное нарушение ограничений системы оценки.

## 9. Политика коммуникации

- Все значимые изменения в правила публикуются через этот репозиторий.
- Раздел часто задаваемых вопросов обновляется по мере накопления типовых вопросов.
- Любое изменение правил должно быть зафиксировано коммитом.
